# Reconnet & Camouflage — Концепция и план реализации

## Контекст

Две задачи:

1. **Базовый реконнект** — клиент восстанавливает сессию после обрыва соединения (существующая проблема из QWEN.md P0)
2. **Camouflage режим** — сервер периодически принудительно дисконнектит клиента для имитации поведения браузера (защита от государственного DPI — Китай, Иран, потенциально РКН в будущем)

## Зачем буфер при реконнекте

При обрыве туннеля:

```
Браузер → TCP(seq=1000) → TUN сервер → WebSocket → TUN клиент → TCP(сервер 8.8.8.8)
                                      ^^^^^^^^^^ обрыв
                                      пакет потерян
```

- Туннель потерял пакет на 2 секунды
- Удалённый сервер (8.8.8.8) ничего не получил → не шлёт ACK
- TCP ждёт RTO = 1-3 сек → потом 6 сек → потом 12 сек
- Браузер видит **таймаут соединения** через 5-10 секунд
- Для приложения это **обрыв страницы**, не просто «задержка»

**Вывод:** без буфера ретрансмит TCP не спасает для живого трафика (веб, SSH, игры).

## Архитектура буфера

```
                     ┌─────────────────┐
                     │  Reconnect      │
                     │  Buffer (RAM)   │
                     │  ~10MB max      │
                     │  FIFO, TTL 60s  │
                     └────────┬────────┘
                              │
TUN → нормальный поток ──────→│→ WebSocket
                              │
TUN → WebSocket down ────────→┘  → буфер
                              │
Reconnect ───────────────────→ сначала буфер, потом нормальный поток
```

**Параметры:**
- `reconnect_buffer_size`: 10MB (настраивается)
- `reconnect_buffer_ttl`: 60 секунд (пакеты старше — дроп)
- Если буфер переполнен → выталкивать самые старые (drop tail)
- Буфер только в RAM (если сервер упал — пакеты потеряны, это нормально)

## Camouflage режим

### Протокол дисконнекта

```
Сервер → Клиент:
  CONTROL CAMOUFLAGE_DISCONNECT (0x19)
  Payload: reconnect_delay_us (8 байт big-endian)

Клиент → Сервер:
  Закрывает соединение через reconnect_delay_us микросекунд
  Потом реконнектится с передачей SessionID

Если клиент не отключился через X секунд:
  Сервер принудительно обрывает + удаляет сессию
  (защита от сторонних/кривых клиентов)
```

### Режим работы

- **Без маскировки** (default) — обычная работа, реконнект только при обрыве
- **С маскировкой** — сервер периодически принудительно дисконнектит клиента
- Конфиг сервера: `camouflage.enabled: true/false`
- Параметры: `min_delay`, `max_delay`, `min_reconnect_delay`, `max_reconnect_delay`

### Имитация браузера

Браузеры не реконнектятся каждые 5 минут. Они:
- Держат соединение пока страница открыта (минуты-часы)
- Открывают несколько соединений к разным доменам одновременно (6+)

Более правильная имитация (будущее):
- Один основной туннель (держится долго)
- Несколько фоновых соединений (открываются/закрываются рандомно)
- Трафик распределяется между ними

Текущий подход (один туннель + рандомный реконнект) — проще и для РКН/Китая достаточно при наличии остальных слоёв (TLS fingerprint, padding).

## План реализации

### Шаг 1: Базовый реконнект (без буфера)

- [ ] Добавить SessionID в AUTH_RESPONSE (при реконнекте клиент передаёт SessionID)
- [ ] Сервер: принимать re-auth по SessionID → восстанавливать сессию вместо создания новой
- [ ] Клиент: реконнект с передачей SessionID
- [ ] Клиент: пересоздаёт TUN с тем же IP
- [ ] Тест: разрыв → реконнект → ping работает

### Шаг 2: Camouflage режим

- [ ] Новый тип 0x19 CAMOUFLAGE_DISCONNECT
- [ ] Сервер: рандомный таймер, конфиг `camouflage.enabled`
- [ ] Клиент: обработка CAMOUFLAGE_DISCONNECT, реконнект по таймеру
- [ ] Сервер: принудительный сброс если клиент не отключился через таймаут
- [ ] Тест: сервер дисконнектит → клиент реконнектится

### Шаг 3: Reconnect Buffer

- [ ] Буферизация на сервере и клиенте при `SessionReconnecting`
- [ ] Drain буфера при восстановлении соединения (отправка до нормального потока)
- [ ] Тест: потери = 0 при реконнекте
